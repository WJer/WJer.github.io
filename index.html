浏览器-浏览器缓存机制

304状态和200状态

当客户端的浏览器向服务器第一次发送HTTP请求时，服务器在给客户端响应时，在响应上提供一些标记（last-modified或者etag），因为是第一次响应，返回的必然是200状态。

缓存过期时，当客户端第二次向服务器发送HTTP请求，该请求头中会有这些标记，服务器会利用这些标记进行比对，如果文件没有被修改，则返回304状态，告诉浏览器上一次的文件还可以使用（它们是留存在cache中的）；如果被修改了，则返回最新的文件信息，返回的是200状态。

两种缓存

协商缓存：会和服务器商量，看看文件是否修改过，缓存的内容时时保持在最新状态。
彻底缓存：在到期之间都不会和服务器交互。即使文件内容发生改变，加载网页还在使用缓存中旧的内容。

last-modified和etag都属于协商缓存，expires和cache-control属于彻底缓存。

详解两种标记

last-modified
这个标记在服务器响应客户端时提供，格式是这样的：

last-modifeid：时间（精确到秒）

如果缓存到期，客户端第二次请求相同的url时，请求头便会带上这个时间信息，但是格式会发生一些变化：

if-modified-since：时间点（和上面的时间是一样的）

服务器收到这个事件信息，如果在这个事件之后，文件有更新，那么发送新的文件，返回200状态，否则返回304状态。

etag
区别于last-modified标志，它的格式是这样的：

Etag:“5d8c72a5edda8d6a:3239″

如果缓存到期，客户端第二次请求相同的url时，请求便会带上这个信息，但是格式会发生一些变化：

If-None-Match:“5d8c72a5edda8d6a:3239″

服务器进行比对，如果不一致，则发送新的文件，返回200状态，否则返回304状态。

为什么会出现etag？
1.Last-Modified标注的最后修改只能精确到秒级，如果某些文件在1秒钟以内，被修改多次的话，它将不能准确标注文件的修改时间
2.如果某些文件会被定期生成，当有时内容并没有任何变化，但Last-Modified却改变了，导致文件没法使用缓存
3.有可能存在服务器没有准确获取文件修改时间，或者与代理服务器时间不一致等情形
（from 百度上某大牛的总结）

ps:两种标记可以一起使用

js-概念小题(1)

第一题

if (!("a" in window)) {
        var a = 1;
}
alert (a);

输出结果：underfined
分析原因：因为预编译的原因，a变量的定义会比if语句先发生，但不会进行赋值操作，所以是underfined

这道题，我还不晓得预编译，蒙也能蒙对。

第二题

var a = 1,
        b = function a (x) {
            x && a (--x);
        };
alert (a);

输出结果：1
分析原因：这样创建函数的方式并不是函数声明方式，a函数并不是全局的。

第三题

function a (x) {
        return x * 2;
}
var a;
alert (a);

输出结果：function….
分析原因：后定义的变量a并没有进行赋值，所以a还是原来的函数，并不会覆盖，所以…..

第四题

function b (x, y, a) {
        arguments[2] = 10;
        alert (a);
}
b(1, 2, 3);

输出结果：10
分析原因：参数会和方法的arguments属性保持同步，一个变，另外一个也会跟着变。

第五题

function a () {
        alert (this);
}
a.call (null);

输出结果：window
分析原因：call()第一个参数为函数执行环境，如果参数是null，会和不传递任何参数有同样的结果，函数的执行环境都是全局的。

这些题在实际开发中并没有什么卵用，放出来只是帮助你深刻了解js的一些概念而已。

js-客户端检测

客户端检测

客户端检测分为三种检测，分别是能力检测，怪癖检测以及用户代理检测。

ps：优先使用能力检测，其次考虑怪癖检测，最后再是用户代理检测。

ps：客户端检测是一种补救措施，不到万不得已，不要使用客户端检测。尽量找到最通用的方案，也就是说各大浏览器都可以识别并正常运行的代码。

能力检测

能力检测不是在识别浏览器，而是识别浏览器的能力，具体的指某个方法支不支持或者某个属性具不具备。
两个重要的概念：

1.先检测达成目的最常用的特性

function getElement(id){
    if(document.getElementById){
        return document.getElementById(id);
    }else if(document.all){
        return document.all[id];
    }else{
        throw new Error("没有这样的方法");
    }
}//IE5之前不支持document.getElementById，但是IE6+、chrome、safari等其他浏览器都支持，所以它被认为是最常用的特性。

2.测试实际要用到的特性。就是说，你要用哪个方法，就测那个方法存不存在。存在的话提供一套方案，不存在的话提供另外一套方案。

function getWindowWidth(){
    if(document.all){
        return document.documentElement.clientWidth;//错误的用法
    }else{
        return window.innerWidth;
    }
}//document.all存在并不意味着clientWidth也存在

能力检测并不是浏览器检测
ps：能力检测并不能确定浏览器。

ps：当需要使用某些特定的浏览器特性时，最好时一次性检测所有的特性，而不要分开检测。

怪癖检测

怪癖检测时想要检测浏览器存在哪些bug。

ps：最好在一开始就执行此类检测。

例子：

var hasEnumShadowsQuick(){
    var o={toString:function(){}};
    var count=o;
    for(var p in o){
        if(p == "toString"){
            count++;
        }
    }
    return (count > 1);
}();

如果是在safari3以前的版本下运行会返回true，因为他会枚举被隐藏的属性。

ps:另外一种检测怪癖的方式是通过try-catch。

用户代理检测

用户代理字符串 

var ua=navigator.userAgent;//获取用户代理字符串

用户代理字符串提供的信息：平台、呈现引擎、版本号（浏览器、内核、Mozilla）等。

浏览器及内核

IE ———— Trident内核
FF ———— Gecko内核
Safari、Chrome ———– Webkit内核
Konqueror ———– KTHML内核
Opera ———– Gecko内核

用户代理字符串技术

识别呈现引擎–>识别浏览器–>识别平台–>识别Windows操作系统–>识别移动设备–>识别游戏系统

js-闭包

闭包

闭包就是指有权访问另一个函数作用域中的变量的函数。

执行环境和作用域

执行环境定义了变量或函数有权访问的其他数据。

每个执行环境都有一个对应的变量对象，eg：window对象，环境内声明的变量和方法都保存在这个对象中。

函数执行时都会创建自己的执行环境及相应的作用域链。

作用域链本质上是一个变量对象的指针列表。函数在访问某一个变量时就会从作用域链一层一层的寻找。

var name="china";
function provice(){
    var name="anhui";
    function city(){
        var name="hefei";
    }
}

分析下这个简单的例子：
1.三个执行环境：全局执行环境，provice()执行环境，city()执行环境
2.city()函数执行时会创建自己的作用域链，依次是city()的活动对象，provice()的活动对象，window对象。
3.provice()函数执行时会创建自己的作用域链，依次是provice()的活动对象，window对象。
4.查找变量时就是按照从最里面到最外面的作用域链访问变量。
5.当city()执行完时，这个函数的执行环境就会退出环境栈，并且对应的变量对象也会相应的销毁。同理 provice()也是。

闭包

function a(){
    var name=1;
    return function (){
        var age=2;
    }
}
var b=a();//b的作用域链包括a函数的活动对象
b=null;//匿名函数不再有引用，垃圾回收机制会安全销毁b函数的作用域链。

以上看来，在a函数的外部可以访问a内部声明的变量和函数。

闭包的问题

问题例子：

function createF(){
    var result=new Array();
    for(var i=0;i<=5;i++){
        result[i]=function(){
            return i;
        }
    }
}
var fs=createF();
for(var i=0;i<=5;i++){
    console.log(fs[i]());//输出5个5
}

本来每个函数都输出自己的i，结果全都输出相同的数。
原因：函数数组中每个函数的作用域链都保存着createF()函数的活动对象，所以它们访问的是同一个i。

解决方式：

function createF(){
    var result=new Array();
    for(var i=0;i<=5;i++){
        result[i]=function(num){
            return function(){//又创建了一个闭包
                return num;//每个函数都有自己的num变量
            }
        }(i);
    }
    return result；
}

js-继承

继承

javascript是通过原型链来实现继承。
实现继承的方式有很多，但总的思路就是原型链，本人总结了一下，如下：

原型链

function Super(){
    ......
}
Super.prototype.saySupername=function(){
    ......
}//此处代表对超类型的原型的操作
function Sub(){
    ......
}
Sub.prototype=new Super();

通过上面的方式，便实现了继承。

注意：
1.这种方式的原型链在最顶层还有一个Object.prototype，不要忽略。
2.可以通过instanceof和isPrototypeOf来确定实例和原型的关系
3.存在的问题：原型中的值会被共享、在创建子类型的实例时无法向超类型的构造函数传递参数。

借用构造函数

function Super(){
    ......
}
function Sub(){
    ......
    Super.call(this,[参数]);//超类型的构造函数当普通函数使用
}

注意：
1.这种方式可以在创建子类型的实例时给超类型的构造函数传递参数。
2.存在的问题：当只借用构造函数这种方式时，那么就无法做到函数复用，除非使用原型。

组合继承

function Super(name){
    this.name=name;
}
Super.prototype.sayName=function(){
    return this.name;
}
function Sub(name,age){
    Super.call(this,name);
    this.age=age;
}
Sub.prototype=new Super();

就是把原型链和借用构造函数两种方式组合起来使用。
注意：1.这是最常用到的继承模式。

原型式继承

function object(o){
    function F(){};
    F.prototype=o;
    return new F();
}

注意：
1.使用Object.create(o);来代替上面的函数，这是ECMAScript5规范了的方法。
2.使用条件：只想使两个对象保持类似的问题。
3.存在的问题：包含引用类型的属性都会共享相应的值，也就是无法有私有的属性和方法。

寄生式继承

function createAnother(o){
    var clone=object(o);
    clone.sayHi=function(){//这里可以创建一些私有的属性和方法
        ......
    }
    return clone;
}

注意：
1.和构造函数一样，若只是使用这种方式，那么就无法达到函数复用。

寄生组合式继承

先看看组合继承的缺点：重复执行相同的函数两次

function Super(name){
    this.name=name;
    this.color=[1,2,3];
}
Super.prototype.sayName=function(){
    console.log(this.name);
}
function Sub(name,age){
    Super.call(this,name);//执行第一次
    this.age=age;
}
Sub.prototype=new Super();//执行第二次

改变后：

function inheritPrototype(Sub,Super){
    var prototype=object(Super.prototype);
    prototype.constructor=Sub;
    Sub.prototype=prototype;
}
function Super(name){
    this.name=name;
    this.color=[1,2,3];
}
Super.prototype.sayName=function(){
    console.log(this.name);
}
function Sub(name,age){
    Super.call(this,name);//执行第一次
    this.age=age;
}
inheritPrototype(Sub,Sup);

注意：
普遍认为这是最理想的继承方法。
